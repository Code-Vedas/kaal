# FAQ / Troubleshooting

## Can I use it without a framework?

Yes. `kaal` is plain Ruby.

## How do I trigger one run manually?

```bash
bundle exec kaal tick
```

## How do I inspect current config and registered jobs?

```bash
bundle exec kaal status
```

## Which backend should I use?

- `memory` for local development
- `redis` for distributed lock coordination
- `sqlite` for simple SQL-backed installs
- `postgres` or `mysql` when you want advisory/named locks plus SQL persistence

## What exactly does Kaal guarantee?

Kaal guarantees at-most-once dispatch per `(key, fire_time)` for Redis, Postgres, and MySQL-backed deployments under the documented crash-and-restart model. See [At-Most-Once Dispatch Guarantee](/dispatch-guarantee).

## Does Kaal guarantee exactly-once side effects?

No. Kaal guarantees dispatch semantics. Use the provided `idempotency_key` in your own database, queue, or API boundary when the job must make downstream effects effectively once.

## Why are jobs dispatching more than once?

Check:

- you are using `redis`, `postgres`, or `mysql` for the shared multi-node backend
- `enable_log_dispatch_registry = true`
- `lease_ttl` is at least `window_lookback + tick_interval`
- each job key is unique
- all nodes use the same namespace and scheduler definition set

See [At-Most-Once Dispatch Guarantee](/dispatch-guarantee) for the full guarantee model.
